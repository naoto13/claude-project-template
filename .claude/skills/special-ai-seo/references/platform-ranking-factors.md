# 各AIプラットフォームのソース選択基準

AIの検索プラットフォームはそれぞれ独自の検索インデックス、ランキングロジック、コンテンツの好みを持つ。このガイドでは、各プラットフォームで引用されるために何が重要かを解説する。

引用元: プリンストン大学のGEO研究（KDD 2024）、SE Rankingドメインオーソリティ研究、ZipTieコンテンツ・回答フィット分析。

---

## 基本事項

すべてのAIプラットフォームに共通する3つの基本要件:

1. **コンテンツがインデックスに含まれていること** — プラットフォームごとに異なる検索バックエンドを使用する（Google、Bing、Brave、または独自のもの）。インデックスされていなければ引用されない。
2. **コンテンツがクロール可能であること** — AIボットはrobots.txtでのアクセスが必要。ボットをブロックすれば引用が失われる。
3. **コンテンツが抽出可能であること** — AIシステムはページ全体ではなく段落を引用する。明確な構造と自己完結した段落が有利。

この基本を押さえた上で、各プラットフォームが重視するシグナルの違いを理解する。

---

## Google AI Overviews

Google AI OverviewはGoogleの独自インデックスからデータを取得し、E-E-A-Tシグナル（経験・専門性・権威性・信頼性）を強く重視する。おおよそ45%のGoogle検索に表示される。

**Google AI Overviewが他と異なる点:** 従来のSEOシグナル（バックリンク、ページオーソリティ、トピック関連性）はすでに考慮済みだ。追加のAI層では、ソース付きコンテンツと構造化データへの好みが加わる。権威ある引用をコンテンツに含めることはAI Overview可視性と132%の相関があり、権威ある（宣伝的でない）文体はさらに89%の向上をもたらすという研究がある。

**重要な点として、AI Overviewは従来の上位10件をそのまま再利用するわけではない。** AI Overviewのソースのうち、通常のオーガニック結果と重複するのは約15%のみ。従来検索では1ページ目に届かないページでも、構造化データと明確・抽出可能な回答があれば引用される可能性がある。

**注力すべき点:**
- スキーママークアップが最大のレバー — Article、FAQPage、HowTo、ProductスキーマはAI Overviewに構造化コンテキストを提供する（可視性30〜40%向上）
- 強力な内部リンクを持つコンテンツクラスターでトピックオーソリティを構築する
- 単なる主張ではなく、名前とソース付きの引用をコンテンツに含める
- 実際の資格を持つ著者プロフィールが重要 — E-E-A-Tが重視される
- 可能な範囲でGoogleのナレッジグラフに入る（正確なWikipediaエントリが助けになる）
- 「やり方」「〜とは」のクエリパターンを狙う — これらがAI Overviewを最もよく誘発する

## ChatGPT

ChatGPTのウェブ検索はBingベースのインデックスを使う。このデータと学習済み知識を組み合わせて回答を生成し、参照したウェブソースを引用する。

**ChatGPTが他と異なる点:** ドメインオーソリティが他のAIプラットフォームより重要。SE Rankingの129,000ドメイン分析によると、オーソリティと信頼性シグナルが引用の約40%を決定し、コンテンツ品質が約35%、プラットフォームの信頼が25%を占める。被リンクドメイン数が非常に多いサイト（35万以上）は1回の回答あたり平均8.4引用を獲得するが、信頼スコアがわずかに低いサイト（97〜100点に対して91〜96点）は8.4から6引用に下がる。

**鮮度は大きな差別化要因だ。** 過去30日以内に更新されたコンテンツは古いコンテンツより約3.2倍多く引用される。ChatGPTは明らかに最新情報を好む。

**最も重要なシグナルはコンテンツと回答のフィット** — 40万ページのZipTie分析によると、コンテンツのスタイルと構造がChatGPT自身の回答形式にどれだけ一致するかが引用可能性の約55%を占める。これはドメインオーソリティ（12%）やオンページ構造（14%）単独より圧倒的に重要だ。ChatGPTが質問に答えるのと同じように書けば、引用元として選ばれる可能性が高まる。

**ChatGPTが自社サイト以外で参照する場所:** WikipediaはChatGPT全引用の7.8%、Redditは1.8%、Forbesは1.1%を占める。ブランドの公式サイトも頻繁に引用されるが、第三者のメンションも大きな重みを持つ。

**注力すべき点:**
- バックリンクとドメインオーソリティに投資する — 最も強力なベースラインシグナル
- 競合コンテンツは最低でも月1回更新する
- ChatGPTが回答を構造化する方法（会話的・直接的・整理された形式）に合わせてコンテンツを作る
- 名前付きソースの検証可能な統計を含める
- 説明的な見出しによるきれいな見出し階層（H1 > H2 > H3）

## Perplexity

Perplexityは常にクリック可能なリンク付きでソースを引用する、最も透明なAI検索プラットフォームだ。独自インデックスとGoogleのインデックスを組み合わせ、複数のリランキングを行う — 初期の関連性取得、従来のランキング要因スコアリング、そして品質基準を満たさない場合は結果セット全体を破棄するMLベースの品質評価。

**Perplexityが他と異なる点:** 最も「リサーチ志向」のAI検索エンジンであり、引用動作にそれが反映されている。Perplexityは権威あるドメイン（Amazon、GitHub、主要学術サイト）のキュレーションリストを管理しており、これらは固有のランキングブーストを得る。新コンテンツを素早く評価する時間減衰アルゴリズムを使用しており、新興パブリッシャーも引用のチャンスがある。

**Perplexityに特有のコンテンツの好み:**
- **FAQスキーマ（JSON-LD）** — FAQの構造化データを持つページは明らかに多く引用される
- **PDFドキュメント** — 公開アクセス可能なPDF（ホワイトペーパー、調査レポート）が優先される。フォームの後ろに権威あるPDFコンテンツを置いているなら、公開バージョンの作成を検討する
- **公開頻度** — キーワードターゲティングより公開の頻度が重要
- **自己完結した段落** — Perplexityはきれいに抽出できる原子的・意味的に完結した段落を好む

**注力すべき点:**
- robots.txtでPerplexityBotを許可する
- Q&Aコンテンツを持つすべてのページにFAQPageスキーマを実装する
- PDFリソースを公開で提供する（ホワイトペーパー、ガイド、レポート）
- 公開・更新のタイムスタンプ付きのArticleスキーマを追加する
- 単独で回答として機能する明確・自己完結した段落で書く
- 特定のニッチで深いトピックオーソリティを構築する

## Microsoft Copilot

CopilotはMicrosoftのエコシステム全体に組み込まれている — Edge、Windows、Microsoft 365、Bing Search。完全にBingのインデックスに依存するため、Bingがコンテンツをインデックスしていなければ引用できない。

**Copilotが他と異なる点:** Microsoftエコシステムとのつながりが独自の最適化機会を生む。LinkedInとGitHubでのメンションやコンテンツは、他のプラットフォームにはないランキングブーストをもたらす。またCopilotはページ速度をより重視する — 2秒未満のロード時間が明確な閾値になっている。

**注力すべき点:**
- Bing Webmaster Toolsにサイトを登録する（多くのサイトはGoogle Search Consoleしか登録していない）
- 新規・更新コンテンツを素早くインデックスするためにIndexNowプロトコルを使用する
- ページ速度を2秒未満に最適化する
- 明確なエンティティ定義を書く — コンテンツが用語や概念を定義する場合、定義を明示的で抽出可能にする
- LinkedInでの存在感を構築する（記事公開、企業ページ管理）し、関連する場合はGitHubも活用する
- BingbotへのフルクロールアクセスをBingに確保する

## Claude

Claudeはウェブ検索が有効な場合、GoogleでもBingでもなくBrave Searchをバックエンドとして使用する。これは全く異なるインデックスであり、Brave Search内での可視性がClaudeがコンテンツを見つけて引用できるかを直接決定する。

**Claudeが他と異なる点:** Claudeは引用する内容について極めて選択的だ。大量のコンテンツを処理するが、引用率は非常に低い — 特定のトピックについて最も事実に正確でソースが充実したコンテンツを探している。具体的な数値と明確な帰属情報を持つデータリッチなコンテンツは、汎用コンテンツよりはるかに優れたパフォーマンスを示す。

**注力すべき点:**
- Brave SearchでコンテンツがBrave検索結果に表示されるか確認する（search.brave.comで自社ブランドと主要キーワードを検索する）
- robots.txtでClaudeBotとanthropicのユーザーエージェントを許可する
- 事実密度を最大化する — 具体的な数値、名前付きソース、日付付き統計
- 説明的な見出しによる明確で抽出可能な構造を使う
- コンテンツ内で権威あるソースを引用する
- トピックについて最も事実に正確なソースを目指す — Claudeは精度を重視する

## robots.txtでAIボットを許可する

robots.txtがAIボットをブロックすると、そのプラットフォームはコンテンツを引用できない。許可すべきユーザーエージェントは以下の通り:

```
User-agent: GPTBot           # OpenAI — ChatGPT検索を動かす
User-agent: ChatGPT-User     # ChatGPTブラウジングモード
User-agent: PerplexityBot    # Perplexity AI検索
User-agent: ClaudeBot        # Anthropic Claude
User-agent: anthropic-ai     # Anthropic Claude（代替）
User-agent: Google-Extended   # Google GeminiおよびAI Overviews
User-agent: Bingbot          # Microsoft Copilot（Bing経由）
Allow: /
```

**学習と検索の違い:** 一部のAIボットはモデルの学習と検索引用の両方に使われる。引用されたいがコンテンツを学習には使わせたくない場合、選択肢は限られる — GPTBotはOpenAIにとって両方を兼ねている。ただし**CCBot**（Common Crawl）はAI検索引用には使われず学習データ収集のみのため、AI検索への影響なくブロックできる。

## どこから始めるか

AI検索への最適化を初めて行う場合、実際にオーディエンスがいるプラットフォームに注力する:

**まずGoogle AI Overviews** — 最も多くのユーザーに届き（Google検索の45%以上）、Google SEOの基盤がすでにある可能性が高い。スキーママークアップを追加し、コンテンツに引用付きのソースを含め、E-E-A-Tシグナルを強化する。

**次にChatGPT** — テックとビジネスオーディエンスが最も多く使うスタンドアロンのAI検索ツール。鮮度（コンテンツを月次で更新）、ドメインオーソリティ、ChatGPTが回答を構成する方法に合わせたコンテンツ構造に注力する。

**その後Perplexity** — リサーチャー、アーリーアダプター、技術専門家がオーディエンスに含まれる場合に特に有効。FAQスキーマを追加し、PDFリソースを公開し、明確で自己完結した段落で書く。

**CopilotとClaudeは優先度が低い**。ただしオーディエンスがエンタープライズ/Microsoft寄り（Copilot）または開発者/アナリスト寄り（Claude）でない限り。それでも基本事項 — 構造化コンテンツ、ソース付きの引用、スキーママークアップ — はすべてのプラットフォームで効果がある。

**すべてのプラットフォームに効果があるアクション:**
1. robots.txtですべてのAIボットを許可する
2. スキーママークアップを実装する（最低でもFAQPage、Article、Organization）
3. コンテンツに名前付きソースの統計データを含める
4. コンテンツを定期的に更新する — 競合トピックは月次で
5. 明確な見出し構造を使う（H1 > H2 > H3）
6. ページ読み込み時間を2秒未満に抑える
7. 資格付きの著者プロフィールを追加する
