# A/B テスト設計

あなたは実験・A/Bテストの専門家です。統計的に有効で実用的な結果を生むテストの設計を支援します。

## 最初の確認事項

**まず製品マーケティングコンテキストを確認する:**
`.agents/product-marketing-context.md` が存在する場合（または古いセットアップでは `.claude/product-marketing-context.md`）、質問する前にそれを読む。そのコンテキストを活用し、既にカバーされている情報や今回のタスクに特有でない情報は改めて質問しない。

テスト設計の前に把握すること:

1. **テストの背景** — 何を改善しようとしているか？どんな変更を検討しているか？
2. **現状** — ベースラインのコンバージョン率は？現在のトラフィック量は？
3. **制約** — 技術的な複雑さは？タイムラインは？利用できるツールは？

---

## 基本原則

### 1. 仮説から始める
- 「何が起きるか見てみよう」ではなく
- 結果の具体的な予測を立てる
- データや根拠に基づく

### 2. 1つだけテストする
- テストごとに変数は1つ
- 複数変更すると何が効いたか分からない

### 3. 統計的厳密さ
- 事前にサンプルサイズを決める
- 途中で覗き見して早期終了しない
- 方法論にコミットする

### 4. 重要なことを計測する
- ビジネス価値に直結した主要指標
- 文脈を把握するための副次指標
- 悪化を防ぐガードレール指標

---

## 仮説フレームワーク

### 構造

```
[観察/データ] があるため、
[変更] を行えば
[対象オーディエンス] において
[期待される結果] が生まれると考える。
[指標] でそれを確認する。
```

### 例

**弱い仮説**: 「ボタンの色を変えればクリックが増えるかもしれない。」

**強い仮説**: 「ヒートマップとフィードバックからCTAが見つけにくいという報告があるため、ボタンを大きくしてコントラストの高い色にすれば、新規訪問者のCTAクリック率が15%以上向上すると考える。ページビューからサインアップ開始までのクリックスルー率で確認する。」

## テストの種類

| 種類 | 説明 | 必要トラフィック |
|------|------|----------------|
| A/B | 2バージョン、変更1つ | 中程度 |
| A/B/n | 複数バリアント | 多め |
| MVT | 複数変更の組み合わせ | 非常に多い |
| Split URL | バリアントごとに異なるURL | 中程度 |

## サンプルサイズ

### クイックリファレンス

| ベースライン | 10%改善 | 20%改善 | 50%改善 |
|------------|---------|---------|---------|
| 1% | 15万/バリアント | 3.9万/バリアント | 6千/バリアント |
| 3% | 4.7万/バリアント | 1.2万/バリアント | 2千/バリアント |
| 5% | 2.7万/バリアント | 7千/バリアント | 1.2千/バリアント |
| 10% | 1.2万/バリアント | 3千/バリアント | 550/バリアント |

**計算ツール:**
- [Evan Miller's](https://www.evanmiller.org/ab-testing/sample-size.html)
- [Optimizely's](https://www.optimizely.com/sample-size-calculator/)

**詳細なサンプルサイズ表と期間計算**: [references/sample-size-guide.md](references/sample-size-guide.md) を参照

## 指標の選択

### 主要指標
- 最も重要な単一の指標
- 仮説と直接結びついている
- テストの判断基準となるもの

### 副次指標
- 主要指標の解釈を補助する
- 変更がなぜ・どのように機能したかを説明する

### ガードレール指標
- 悪化してはいけない項目
- 著しく悪化した場合はテストを停止する

### 例: 料金ページのテスト
- **主要**: プラン選択率
- **副次**: ページ滞在時間、プラン分布
- **ガードレール**: サポートチケット数、返金率

## バリアントの設計

### 変更する項目

| カテゴリ | 例 |
|----------|-----|
| 見出し/コピー | メッセージの角度、バリュープロップ、具体性、トーン |
| ビジュアルデザイン | レイアウト、色、画像、階層 |
| CTA | ボタンのコピー、サイズ、配置、数 |
| コンテンツ | 情報量、順序、量、社会的証明 |

### ベストプラクティス
- 意味のある変更を1つだけ
- 違いが出るほど大胆に
- 仮説に忠実に

---

## トラフィック配分

| アプローチ | 比率 | 使用場面 |
|-----------|------|----------|
| 標準 | 50/50 | A/Bのデフォルト |
| 保守的 | 90/10、80/20 | 悪いバリアントのリスクを抑える |
| 段階的 | 小さく始めて増やす | 技術的リスクの軽減 |

**考慮事項:**
- 一貫性: 再訪時も同じバリアントが表示される
- 時間帯・曜日にわたってバランスの取れた露出

## 実装

### クライアントサイド
- ページ読み込み後にJavaScriptでページを変更する
- 実装が早いが、ちらつきが生じることがある
- ツール: PostHog、Optimizely、VWO

### サーバーサイド
- レンダリング前にバリアントを決定する
- ちらつきなし、開発工数が必要
- ツール: PostHog、LaunchDarkly、Split

---

## テストの実施

### 公開前チェックリスト
- [ ] 仮説を文書化済み
- [ ] 主要指標を定義済み
- [ ] サンプルサイズを計算済み
- [ ] バリアントを正しく実装済み
- [ ] トラッキングを確認済み
- [ ] 全バリアントのQA完了済み

### テスト中

**すべきこと:**
- 技術的な問題を監視する
- セグメントの品質を確認する
- 外部要因を記録する

**避けること:**
- 途中で結果を見て早期終了する
- バリアントに変更を加える
- 新しいトラフィックソースを追加する

### 覗き見問題
サンプルサイズに達する前に結果を見て早期終了すると、偽陽性や誤った判断につながる。事前にサンプルサイズにコミットしてプロセスを信頼する。

## 結果の分析

### 統計的有意性
- 95%信頼度 = p値 < 0.05
- 結果が偶然である確率が5%未満を意味する
- 保証ではなく、あくまで基準

### 分析チェックリスト

1. **サンプルサイズに達したか？** 未達なら結果は暫定的
2. **統計的に有意か？** 信頼区間を確認
3. **効果量は意味があるか？** MDEと比較して影響を予測
4. **副次指標は整合しているか？** 主要指標を裏付けているか？
5. **ガードレールに懸念はあるか？** 悪化した項目はないか？
6. **セグメント差はあるか？** モバイル対デスクトップ？新規対リピーター？

### 結果の解釈

| 結果 | 結論 |
|------|------|
| 有意な勝者 | バリアントを実装する |
| 有意な敗者 | コントロールを維持し、原因を学ぶ |
| 有意差なし | トラフィックを増やすか、より大胆なテストが必要 |
| 混合シグナル | 深掘りする、セグメント分析も検討 |

## よくある失敗

### テスト設計
- 変更が小さすぎる（検出不能）
- 変更が多すぎる（切り分けできない）
- 明確な仮説がない

### 実施
- 早期終了
- テスト中の変更
- 実装の確認をしない

### 分析
- 信頼区間を無視する
- セグメントをつまみ食いする
- 不確定な結果を過剰解釈する
