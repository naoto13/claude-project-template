# スキルクリエーター

このスキルは、効果的なスキルを作成するためのガイダンスを提供します。

## スキルとは

スキルはモジュール式の自己完結型パッケージで、特化した知識・ワークフロー・ツールを提供することで Claude の能力を拡張します。特定のドメインやタスクに対する「オンボーディングガイド」として機能し、Claude を汎用エージェントから、いかなるモデルも完全には持てない手続き的知識を備えた専門エージェントへと変換します。

### スキルが提供するもの

1. 特化したワークフロー — 特定ドメイン向けの複数ステップの手順
2. ツール統合 — 特定のファイル形式や API を扱う手順
3. ドメイン専門知識 — 会社固有の知識・スキーマ・ビジネスロジック
4. バンドルリソース — 複雑かつ反復的なタスク向けのスクリプト・参照資料・アセット

## コア原則

### 簡潔さが鍵

コンテキストウィンドウは公共財です。スキルはコンテキストウィンドウをシステムプロンプト・会話履歴・他のスキルのメタデータ・実際のユーザーリクエストと共有します。

**デフォルトの前提: Claude はすでに非常に賢い。** Claude がまだ持っていないコンテキストのみ追加してください。情報の各ピースに問いかけましょう：「Claude はこの説明を本当に必要としているか？」「この段落はトークンコストに見合うか？」

冗長な説明より簡潔な例を優先してください。

### 適切な自由度の設定

タスクの繊細さと可変性に合わせて具体性のレベルを調整してください：

**高自由度（テキストベースの指示）**: 複数のアプローチが有効、判断がコンテキストに依存する、またはヒューリスティックがアプローチを導く場合に使用。

**中自由度（擬似コードまたはパラメーター付きスクリプト）**: 望ましいパターンが存在し、ある程度の変化が許容される、またはコンフィギュレーションが動作に影響する場合に使用。

**低自由度（具体的なスクリプト、少ないパラメーター）**: 操作が繊細でエラーが起きやすい、一貫性が重要、または特定の順序に従わなければならない場合に使用。

Claude をパスを探索する存在として考えてください：崖のある狭い橋には具体的なガードレール（低自由度）が必要で、広い野原では多くのルートが可能（高自由度）です。

### スキルの構造

すべてのスキルは必須の SKILL.md ファイルとオプションのバンドルリソースで構成されます：

```
skill-name/
├── SKILL.md (必須)
│   ├── YAML frontmatter メタデータ (必須)
│   │   ├── name: (必須)
│   │   └── description: (必須)
│   └── Markdown 指示 (必須)
└── バンドルリソース (任意)
    ├── scripts/          - 実行可能コード (Python/Bash 等)
    ├── references/       - 必要に応じてコンテキストに読み込む文書
    └── assets/           - 出力で使用するファイル (テンプレート、アイコン、フォント等)
```

#### SKILL.md（必須）

すべての SKILL.md は以下で構成されます：

- **Frontmatter**（YAML）: `name` と `description` フィールドを含む。スキルがいつ使用されるかを Claude が判断するために読む唯一のフィールドのため、スキルの内容と使用すべきタイミングを明確かつ網羅的に記述することが重要です。
- **Body**（Markdown）: スキルの使用方法の指示とガイダンス。スキルがトリガーされた後にのみ読み込まれます。

#### バンドルリソース（任意）

##### スクリプト（`scripts/`）

決定論的な信頼性が必要な、または繰り返し書き直されるタスク向けの実行可能コード（Python/Bash 等）。

- **含める場合**: 同じコードが繰り返し書き直される場合、または決定論的な信頼性が必要な場合
- **例**: PDF 回転タスク用の `scripts/rotate_pdf.py`
- **メリット**: トークン効率が良い、決定論的、コンテキストに読み込まずに実行可能
- **注意**: パッチ適用や環境固有の調整のために Claude が読む必要がある場合もあります

##### 参照資料（`references/`）

Claude のプロセスや思考を支援するために必要に応じてコンテキストに読み込む文書・参照資料。

- **含める場合**: Claude が作業中に参照すべき文書がある場合
- **例**: 財務スキーマ用の `references/finance.md`、社内 NDA テンプレートの `references/mnda.md`、社内ポリシーの `references/policies.md`、API 仕様の `references/api_docs.md`
- **用途**: データベーススキーマ、API ドキュメント、ドメイン知識、会社ポリシー、詳細なワークフローガイド
- **メリット**: SKILL.md をスリムに保ち、Claude が必要と判断した場合のみ読み込まれる
- **ベストプラクティス**: ファイルが大きい場合（10k ワード超）は SKILL.md に grep 検索パターンを含める
- **重複を避ける**: 情報は SKILL.md または参照ファイルのどちらか一方に存在すべき。詳細情報は参照ファイルを優先し、SKILL.md をスリムに保つ。SKILL.md には必須の手続き的指示とワークフローガイダンスのみを保持し、詳細な参照資料・スキーマ・例は参照ファイルに移す。

##### アセット（`assets/`）

コンテキストに読み込まれるのではなく、Claude が生成する出力内で使用されるファイル。

- **含める場合**: スキルが最終出力で使用するファイルが必要な場合
- **例**: ブランドアセット用の `assets/logo.png`、PowerPoint テンプレートの `assets/slides.pptx`、HTML/React ボイラープレートの `assets/frontend-template/`、タイポグラフィ用の `assets/font.ttf`
- **用途**: テンプレート、画像、アイコン、ボイラープレートコード、フォント、コピーまたは修正されるサンプルドキュメント
- **メリット**: 出力リソースをドキュメントから分離し、コンテキストに読み込まずにファイルを使用できる

#### スキルに含めるべきでないもの

スキルにはその機能を直接サポートする必須ファイルのみ含めてください。以下のような余分な文書や補助ファイルは作成しないでください：

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- 等

スキルには AI エージェントが目の前の作業をこなすために必要な情報のみ含めるべきです。作成プロセス、セットアップ・テスト手順、ユーザー向けドキュメントなどの補助コンテキストは含めません。追加の文書ファイルを作成しても散乱と混乱を招くだけです。

### 段階的開示のデザイン原則

スキルはコンテキストを効率的に管理するために 3 段階の読み込みシステムを使用します：

1. **メタデータ（name + description）** — 常にコンテキストに存在（約 100 ワード）
2. **SKILL.md 本文** — スキルがトリガーされた時（5k ワード以下）
3. **バンドルリソース** — Claude が必要と判断した場合（スクリプトはコンテキストウィンドウに読み込まずに実行可能なため無制限）

#### 段階的開示のパターン

コンテキストの肥大化を最小限に抑えるために SKILL.md 本文は必須事項のみとし、500 行以下に保ってください。この制限に近づいたらコンテンツを別ファイルに分割します。コンテンツを他のファイルに分割する場合、SKILL.md から参照してそれらの存在と使用タイミングを明確に記述することが非常に重要です。

**主要原則:** スキルが複数のバリエーション・フレームワーク・オプションをサポートする場合、SKILL.md にはコアワークフローと選択ガイダンスのみを保持します。バリアント固有の詳細（パターン、例、コンフィギュレーション）は別の参照ファイルに移します。

**パターン 1: 高レベルガイドと参照**

```markdown
# PDF 処理

## クイックスタート

pdfplumber でテキストを抽出する:
[コード例]

## 高度な機能

- **フォーム入力**: 完全なガイドは [FORMS.md](FORMS.md) を参照
- **API リファレンス**: 全メソッドは [REFERENCE.md](REFERENCE.md) を参照
- **例**: 一般的なパターンは [EXAMPLES.md](EXAMPLES.md) を参照
```

Claude は必要な場合のみ FORMS.md、REFERENCE.md、または EXAMPLES.md を読み込みます。

**パターン 2: ドメイン固有の整理**

複数のドメインを持つスキルでは、無関係なコンテキストの読み込みを避けるためにドメイン別にコンテンツを整理します：

```
bigquery-skill/
├── SKILL.md (概要とナビゲーション)
└── reference/
    ├── finance.md (収益、請求メトリクス)
    ├── sales.md (商談、パイプライン)
    ├── product.md (API 使用状況、機能)
    └── marketing.md (キャンペーン、アトリビューション)
```

ユーザーが売上メトリクスを尋ねた場合、Claude は sales.md のみを読みます。

同様に、複数のフレームワークやバリアントをサポートするスキルでは、バリアント別に整理します：

```
cloud-deploy/
├── SKILL.md (ワークフロー + プロバイダー選択)
└── references/
    ├── aws.md (AWS デプロイパターン)
    ├── gcp.md (GCP デプロイパターン)
    └── azure.md (Azure デプロイパターン)
```

ユーザーが AWS を選択した場合、Claude は aws.md のみを読みます。

**パターン 3: 条件付き詳細**

基本コンテンツを表示し、高度なコンテンツへのリンクを提供します：

```markdown
# DOCX 処理

## ドキュメントの作成

新規ドキュメントには docx-js を使用します。[DOCX-JS.md](DOCX-JS.md) を参照。

## ドキュメントの編集

シンプルな編集には XML を直接修正します。

**変更追跡の場合**: [REDLINING.md](REDLINING.md) を参照
**OOXML の詳細**: [OOXML.md](OOXML.md) を参照
```

Claude はユーザーがそれらの機能を必要とする場合のみ REDLINING.md または OOXML.md を読みます。

**重要なガイドライン:**

- **深いネストの参照を避ける** — 参照は SKILL.md から 1 階層のみ。すべての参照ファイルは SKILL.md から直接リンクします。
- **長い参照ファイルを構造化する** — 100 行を超えるファイルには先頭に目次を含め、Claude がプレビュー時に全体像を把握できるようにします。

## スキル作成プロセス

スキル作成は以下のステップを含みます：

1. 具体的な例でスキルを理解する
2. 再利用可能なスキルコンテンツを計画する（スクリプト、参照資料、アセット）
3. スキルを初期化する（init_skill.py を実行）
4. スキルを編集する（リソースを実装し SKILL.md を作成）
5. スキルをパッケージ化する（package_skill.py を実行）
6. 実際の使用に基づいて反復する

適用できない明確な理由がある場合のみステップをスキップして、これらのステップを順番に従ってください。

### ステップ 1: 具体的な例でスキルを理解する

スキルの使用パターンがすでに明確に理解されている場合のみこのステップをスキップしてください。既存のスキルを扱う場合でも有効です。

効果的なスキルを作成するには、スキルがどのように使用されるかの具体的な例を明確に理解します。この理解はユーザーから直接提供された例、またはユーザーのフィードバックで検証された生成例から得られます。

例えば、image-editor スキルを構築する場合、関連する質問には以下が含まれます：

- 「image-editor スキルはどんな機能をサポートすべきですか？編集、回転、その他は？」
- 「このスキルがどのように使われるか例を挙げてもらえますか？」
- 「ユーザーが『この画像の赤目を除去して』や『この画像を回転して』と聞くことが想像できますが、他にどんな使い方が想定されますか？」
- 「このスキルをトリガーすべきユーザーの発言は？」

ユーザーを圧倒しないよう、1 つのメッセージで多くの質問をしすぎないでください。最も重要な質問から始め、より効果的な結果のために必要に応じてフォローアップしてください。

スキルがサポートすべき機能について明確な感覚が得られたらこのステップを終了します。

### ステップ 2: 再利用可能なスキルコンテンツの計画

具体的な例を効果的なスキルに変えるために、各例を以下のように分析します：

1. ゼロから例を実行する方法を考える
2. これらのワークフローを繰り返し実行する際に役立つスクリプト・参照資料・アセットを特定する

例：「この PDF を回転して」というクエリを処理する `pdf-editor` スキルを構築する場合の分析：

1. PDF の回転には毎回同じコードを書き直す必要がある
2. スキルに保存する `scripts/rotate_pdf.py` スクリプトが役立つ

例：「Todo アプリを作って」や「ステップ数を追跡するダッシュボードを作って」というクエリ向けの `frontend-webapp-builder` スキルを設計する場合の分析：

1. フロントエンドウェブアプリの作成には毎回同じ HTML/React のボイラープレートが必要
2. ボイラープレートの HTML/React プロジェクトファイルを含む `assets/hello-world/` テンプレートがスキルに保存すると役立つ

例：「今日ログインしたユーザーは何人いますか？」というクエリを処理する `big-query` スキルを構築する場合の分析：

1. BigQuery のクエリには毎回テーブルスキーマと関係を再発見する必要がある
2. テーブルスキーマを文書化した `references/schema.md` ファイルがスキルに保存すると役立つ

スキルのコンテンツを確立するために、各具体的な例を分析して含める再利用可能なリソースのリストを作成します：スクリプト、参照資料、アセット。

### ステップ 3: スキルの初期化

ここで実際にスキルを作成します。

スキルがすでに存在し、反復またはパッケージ化が必要な場合のみこのステップをスキップしてください。その場合は次のステップに進みます。

ゼロから新しいスキルを作成する場合、常に `init_skill.py` スクリプトを実行してください。このスクリプトはスキルに必要なすべてを自動的に含む新しいテンプレートスキルディレクトリを生成し、スキル作成プロセスをより効率的で信頼性の高いものにします。

使用方法：

```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

スクリプトの処理内容：

- 指定されたパスにスキルディレクトリを作成
- 適切な frontmatter と TODO プレースホルダーを含む SKILL.md テンプレートを生成
- サンプルリソースディレクトリを作成：`scripts/`、`references/`、`assets/`
- カスタマイズまたは削除可能なサンプルファイルを各ディレクトリに追加

初期化後、生成された SKILL.md とサンプルファイルを必要に応じてカスタマイズまたは削除してください。

### ステップ 4: スキルの編集

（新規生成または既存の）スキルを編集する際、スキルは別の Claude インスタンスが使用するために作成されていることを忘れないでください。Claude にとって有益で自明でない情報を含めてください。別の Claude インスタンスがこれらのタスクをより効果的に実行するために役立つ手続き的知識、ドメイン固有の詳細、または再利用可能なアセットを考慮してください。

#### 実証済みのデザインパターンを学ぶ

スキルのニーズに応じてこれらの参考ガイドを参照してください：

- **複数ステップのプロセス**: 順次ワークフローと条件分岐ロジックについては references/workflows.md を参照
- **特定の出力フォーマットや品質基準**: テンプレートと例のパターンについては references/output-patterns.md を参照

これらのファイルには効果的なスキルデザインのための確立されたベストプラクティスが含まれています。

#### 再利用可能なスキルコンテンツから始める

実装を開始するには、上記で特定した再利用可能なリソースから始めます：`scripts/`、`references/`、`assets/` ファイル。このステップではユーザーの入力が必要な場合があります。例えば、`brand-guidelines` スキルを実装する際、ユーザーは `assets/` に保存するブランドアセットやテンプレート、または `references/` に保存するドキュメントを提供する必要があるかもしれません。

追加したスクリプトは実際に実行してバグがないこと、出力が期待通りであることを確認する必要があります。類似するスクリプトが多い場合、すべてが機能するという確信を得ながら完了までの時間とのバランスを取るために、代表的なサンプルのみをテストする必要があります。

スキルに不要なサンプルファイルとディレクトリは削除してください。初期化スクリプトは構造を示すために `scripts/`、`references/`、`assets/` にサンプルファイルを作成しますが、ほとんどのスキルはそれらすべてを必要としません。

#### SKILL.md を更新する

**作成ガイドライン:** 常に命令形/不定詞形を使用してください。

##### Frontmatter

`name` と `description` を含む YAML frontmatter を作成します：

- `name`: スキル名
- `description`: これがスキルの主要なトリガーメカニズムであり、Claude がスキルをいつ使用するかを理解するのに役立ちます。
  - スキルの機能と使用すべきタイミング・コンテキストの両方を含めてください。
  - 「使用すべき時」の情報はすべてここに記載します。本文には記載しません。本文はトリガー後にのみ読み込まれるため、本文の「このスキルを使用する時」セクションは Claude の助けになりません。
  - `docx` スキルの description 例：「トラック変更・コメント・フォーマット保持・テキスト抽出をサポートする、包括的なドキュメントの作成・編集・分析。Claude が専門文書（.docx ファイル）を扱う必要がある場合に使用：(1) 新規ドキュメントの作成、(2) コンテンツの変更や編集、(3) トラック変更の操作、(4) コメントの追加、またはその他のドキュメントタスク」

YAML frontmatter には他のフィールドを含めないでください。

##### Body

スキルとそのバンドルリソースの使用方法の指示を作成します。

### ステップ 5: スキルのパッケージ化

スキルの開発が完了したら、ユーザーと共有する配布可能な .skill ファイルにパッケージ化する必要があります。パッケージ化プロセスはまずスキルを自動的に検証してすべての要件を満たしていることを確認します：

```bash
scripts/package_skill.py <path/to/skill-folder>
```

出力ディレクトリのオプション指定：

```bash
scripts/package_skill.py <path/to/skill-folder> ./dist
```

パッケージ化スクリプトの処理内容：

1. **検証**: スキルを自動的に検証し、以下を確認します：
   - YAML frontmatter のフォーマットと必須フィールド
   - スキルの命名規則とディレクトリ構造
   - description の完全性と品質
   - ファイルの整理とリソースの参照

2. **パッケージ化**: 検証が通った場合、スキルにちなんで名付けられた .skill ファイル（例：`my-skill.skill`）を作成します。このファイルはすべてのファイルを含み、配布に適した適切なディレクトリ構造を維持します。.skill ファイルは .skill 拡張子を持つ zip ファイルです。

検証が失敗した場合、スクリプトはエラーを報告してパッケージを作成せずに終了します。検証エラーを修正してパッケージ化コマンドを再実行してください。

### ステップ 6: 反復

スキルをテストした後、ユーザーが改善を要求する場合があります。多くの場合、スキルがどのように機能したかの新鮮なコンテキストを持ちながら、スキルを使用した直後に行われます。

**反復ワークフロー:**

1. 実際のタスクでスキルを使用する
2. 苦労や非効率に気づく
3. SKILL.md またはバンドルリソースをどのように更新すべきか特定する
4. 変更を実装して再度テストする
