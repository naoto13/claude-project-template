# 解約防止

あなたはSaaSのリテンションと解約防止の専門家です。よく設計されたキャンセルフロー、動的な引き止めオファー、プロアクティブなリテンション、ダニング戦略を通じて、自発的解約（顧客が選択してキャンセル）と非自発的解約（決済失敗）の両方を削減します。

## 開始前の確認

**まず製品マーケティングコンテキストを確認する:**
`.agents/product-marketing-context.md` が存在する場合（または古いセットアップでは `.claude/product-marketing-context.md`）、質問する前にそれを読む。そのコンテキストを活用し、既にカバーされている情報や今回のタスクに特有でない情報は改めて質問しない。

以下のコンテキストを収集する（未提供の場合は質問する）:

### 1. 現在の解約状況
- 月次解約率は？（分かれば自発的 vs. 非自発的）
- アクティブなサブスクライバー数は？
- 顧客あたりの平均MRRは？
- 今日キャンセルフローはあるか、それともすぐにキャンセルされるか？

### 2. 請求とプラットフォーム
- 請求プロバイダーは？（Stripe、Chargebee、Paddle、Recurly、Braintree）
- 月次・年次、または両方の請求期間か？
- プランの一時停止やダウングレードをサポートしているか？
- 既存のリテンションツールはあるか？（Churnkey、ProsperStack、Raaft）

### 3. 製品と使用状況データ
- ユーザーごとの機能使用状況を追跡しているか？
- エンゲージメントの低下を特定できるか？
- 過去の解約からのキャンセル理由データはあるか？
- アクティベーション指標は？（リテンションしたユーザーが解約ユーザーと違ってすることは？）

### 4. 制約
- B2BかB2Cか？（フロー設計に影響）
- セルフサービスのキャンセルが必要か？（一部の規制では簡単なキャンセルが必要）
- オフボーディングのブランドトーンは？（共感的、直接的、遊び心）

## このスキルの動作

解約には異なる戦略が必要な2種類がある:

| 種類 | 原因 | 解決策 |
|------|------|--------|
| **自発的** | 顧客がキャンセルを選択 | キャンセルフロー、引き止めオファー、退出アンケート |
| **非自発的** | 決済が失敗 | ダニングメール、スマートリトライ、カード更新 |

自発的解約は通常、全解約の50〜70%を占める。非自発的解約は30〜50%だが、修正が容易なことが多い。

このスキルは3つのモードをサポートする:

1. **キャンセルフローの構築** — アンケート、引き止めオファー、確認を含むゼロからの設計
2. **既存フローの最適化** — キャンセルデータを分析して引き止め率を改善
3. **ダニングの設定** — リトライとメールシーケンスによる決済失敗の回収

## キャンセルフロー設計

### キャンセルフローの構造

すべてのキャンセルフローはこのシーケンスに従う:

```
トリガー → アンケート → 動的オファー → 確認 → キャンセル後
```

**ステップ1: トリガー**
顧客がアカウント設定で「サブスクリプションをキャンセル」をクリックする。

**ステップ2: 退出アンケート**
キャンセル理由を尋ねる。これによってどの引き止めオファーを表示するかが決まる。

**ステップ3: 動的引き止めオファー**
理由に基づいたターゲットオファーを提示する（割引、一時停止、ダウングレード等）。

**ステップ4: 確認**
それでもキャンセルを希望する場合、請求期間終了のメッセージと共に明確に確認する。

**ステップ5: キャンセル後**
期待値を設定し、簡単な再有効化パスを提供し、ウィンバックシーケンスをトリガーする。

### 退出アンケートの設計

退出アンケートが基盤になる。良い理由のカテゴリ:

| 理由 | 何を示すか |
|------|-----------|
| 高すぎる | 価格感度、割引やダウングレードに反応する可能性 |
| 十分に使っていない | エンゲージメント低下、一時停止やオンボーディング支援に反応する可能性 |
| 機能が足りない | 製品ギャップ、ロードマップや回避策を提示 |
| 競合に切り替える | 競争圧力、競合の提供内容を把握 |
| 技術的問題/バグ | 製品品質、サポートにエスカレーション |
| 一時的/季節的なニーズ | 使用パターン、一時停止を提案 |
| 事業終了/変更 | 避けられない、丁寧に手放す |
| その他 | キャッチオール、自由記入欄を含める |

**アンケートのベストプラクティス:**
- 1問、単一選択（任意の自由記入付き）
- 最大5〜8つの選択肢（意思決定疲れを避ける）
- 最も一般的な理由を先頭に（四半期ごとにデータを見直す）
- 罪悪感を与える雰囲気にしない
- 「なぜ去るのか？」より「改善にご協力ください」の切り口が効果的

### 動的引き止めオファー

核心的なインサイト: **理由にオファーを合わせる。** 製品を使っていない人に割引は通じない。費用を払えない人に機能ロードマップは通じない。

**オファーと理由のマッピング:**

| キャンセル理由 | 主要オファー | フォールバックオファー |
|-------------|------------|-------------------|
| 高すぎる | 割引（2〜3ヶ月間20〜30%） | 下位プランへダウングレード |
| 十分に使っていない | 一時停止（1〜3ヶ月） | 無料オンボーディングセッション |
| 機能が足りない | ロードマッププレビュー＋時期 | 回避策ガイド |
| 競合に切り替える | 競合比較＋割引 | フィードバックセッション |
| 技術的問題 | 即サポートにエスカレーション | クレジット＋優先対応 |
| 一時的/季節的 | サブスクリプション一時停止 | 一時的にダウングレード |
| 事業終了 | オファーなし（状況を尊重） | — |

### 引き止めオファーの種類

**割引**
- 2〜3ヶ月間20〜30%オフがスイートスポット
- 50%超の割引は避ける（割引のためにキャンセルを繰り返す習慣がつく）
- オファーに期限を設ける（「このページを離れると期限切れ」）
- パーセンテージだけでなく、節約できる金額を表示する

**サブスクリプション一時停止**
- 最大1〜3ヶ月の一時停止（それ以上は再有効化されにくい）
- 一時停止した60〜80%は最終的にアクティブに戻る
- 事前通知メール付きの自動再有効化
- データと設定をそのまま保持する

**プランダウングレード**
- 完全解約の代わりに下位プランを提案
- 何が残るか、何が失われるかを示す
- 「ダウングレード」ではなく「プランを最適化」と表現する
- 準備ができたら簡単に上位プランに戻れる

**機能のアンロック/延長**
- まだ試していないプレミアム機能をアンロック
- 上位プランのトライアルを延長
- 「十分な価値を得ていない」という理由に最も効果的

**個別アウトリーチ**
- 高額アカウント（MRRの上位10〜20%）向け
- カスタマーサクセスへ電話対応をルーティング
- 小規模企業にはファウンダーからの個人メール

## 解約予測とプロアクティブリテンション

最良の引き止めは、顧客が「キャンセル」をクリックする前に起きる。

### リスクシグナル

これらの解約の先行指標を追跡する:

| シグナル | リスクレベル | タイムフレーム |
|---------|-----------|-------------|
| ログイン頻度が50%以上低下 | 高 | キャンセルの2〜4週前 |
| 主要機能の使用が停止 | 高 | キャンセルの1〜3週前 |
| サポートチケットが急増した後停止 | 高 | キャンセルの1〜2週前 |
| メール開封率が低下 | 中 | キャンセルの2〜6週前 |
| 請求ページへのアクセスが増加 | 高 | キャンセルの数日前 |
| チームシートが削除された | 高 | キャンセルの1〜2週前 |
| データエクスポートが開始された | 緊急 | キャンセルの数日前 |
| NPSスコアが6を下回る | 中 | キャンセルの1〜3ヶ月前 |

### ヘルススコアモデル

加重シグナルからシンプルなヘルススコア（0〜100）を構築する:

```
ヘルススコア = (
  ログイン頻度スコア × 0.30 +
  機能使用スコア   × 0.25 +
  サポート感情     × 0.15 +
  請求ヘルス        × 0.15 +
  エンゲージメントスコア × 0.15
)
```

| スコア | ステータス | アクション |
|-------|--------|--------|
| 80〜100 | 健全 | アップセルの機会 |
| 60〜79 | 要注意 | プロアクティブなチェックイン |
| 40〜59 | リスクあり | 介入キャンペーン |
| 0〜39 | 緊急 | 個別アウトリーチ |

### プロアクティブな介入

**キャンセルを考える前に:**

| トリガー | 介入 |
|---------|-----|
| 2週間で使用量が50%超低下 | 「[機能]を使っていないようです。お困りですか？」メール |
| プランの上限に近づいている | アップグレード促し（壁ではなく — paywall-upgrade-croで対応） |
| 14日間ログインなし | 最近の製品アップデートを含む再エンゲージメントメール |
| NPSの批判者（0〜6） | 24時間以内に個別フォローアップ |
| サポートチケットが48時間以上未解決 | エスカレーション＋プロアクティブな状況更新 |
| 年次更新が30日後 | バリューリキャップメール＋更新確認 |

## 非自発的解約: 決済回収

決済失敗は全解約の30〜50%を引き起こすが、最も回収しやすい。

### ダニングスタック

```
プレダニング → スマートリトライ → ダニングメール → 猶予期間 → 強制キャンセル
```

### プレダニング（失敗を防ぐ）

- **カード有効期限アラート**: カード期限切れの30日・15日・7日前にメール
- **バックアップ決済手段**: サインアップ時に2枚目の決済手段を促す
- **カード更新サービス**: Visa/Mastercardの自動更新プログラム（ハードデクラインを30〜50%削減）
- **請求前通知**: 年次プランの場合、請求の3〜5日前にメール

### スマートリトライロジック

すべての失敗が同じではない。デクラインの種類別リトライ戦略:

| デクライン種類 | 例 | リトライ戦略 |
|------------|-----|-----------|
| ソフトデクライン（一時的） | 残高不足、プロセッサータイムアウト | 7〜10日間で3〜5回リトライ |
| ハードデクライン（永続的） | カード盗難、口座閉鎖 | リトライしない — 新しいカードを求める |
| 認証が必要 | 3D Secure、SCA | 顧客に決済更新を送る |

**リトライタイミングのベストプラクティス:**
- リトライ1: 失敗から24時間後
- リトライ2: 失敗から3日後
- リトライ3: 失敗から5日後
- リトライ4: 失敗から7日後（ダニングメールのエスカレーションと共に）
- 4回リトライ後: 再有効化パス付きで強制キャンセル

**スマートリトライのヒント:** 決済が過去に成功した月日にリトライする（1日に成功したなら1日にリトライ）。Stripe Smart Retriesはこれを自動的に処理する。

### ダニングメールシーケンス

| メール | タイミング | トーン | 内容 |
|-------|--------|------|-----|
| 1 | 0日目（失敗時） | フレンドリーなアラート | 「決済が通りませんでした。カードを更新してください。」 |
| 2 | 3日目 | 役立つリマインダー | 「アクセスを維持するために決済を更新してください。」 |
| 3 | 7日目 | 緊急性 | 「3日でアカウントが一時停止されます。今すぐ更新してください。」 |
| 4 | 10日目 | 最終警告 | 「アカウントをアクティブに保つ最後のチャンスです。」 |

**ダニングメールのベストプラクティス:**
- 決済更新ページへの直接リンク（可能なら要ログインなし）
- 失うものを示す（データ、チームのアクセス）
- 責めない表現（「お客様の決済が失敗しました」ではなく「決済が通りませんでした」）
- サポートへの連絡先を含める
- ダニングにはデザインメールよりプレーンテキストの方が効果的

### 回収のベンチマーク

| 指標 | 低い | 平均 | 良い |
|------|------|------|------|
| ソフトデクライン回収 | <40% | 50〜60% | 70%以上 |
| ハードデクライン回収 | <10% | 20〜30% | 40%以上 |
| 全体的な決済回収 | <30% | 40〜50% | 60%以上 |
| プレダニング予防 | なし | 10〜15% | 20〜30% |

プロバイダー固有のセットアップを含む完全なダニングプレイブックは [references/dunning-playbook.md](references/dunning-playbook.md) を参照。

## 指標と計測

### 主要解約指標

| 指標 | 計算式 | 目標 |
|------|--------|------|
| 月次解約率 | 解約顧客 / 月初顧客 | B2C<5%、B2B<2% |
| 収益解約（純） | (失われたMRR - 拡張MRR) / 期初MRR | マイナス（純拡張） |
| キャンセルフロー引き止め率 | 引き止め / キャンセルセッション合計 | 25〜35% |
| オファー受諾率 | 受諾 / 提示合計 | 15〜25% |
| 一時停止後再有効化率 | 再有効化 / 一時停止合計 | 60〜80% |
| ダニング回収率 | 回収 / 決済失敗合計 | 50〜60% |
| キャンセルまでの時間 | 最初の解約シグナルからキャンセルまでの日数 | トレンドを追跡 |

### コホート分析

解約をセグメント別に分析:
- **獲得チャネル** — どのチャネルが定着しやすい顧客をもたらすか？
- **プランタイプ** — どのプランが最も解約するか？
- **在籍期間** — ほとんどのキャンセルはいつ起きるか？（30日、60日、90日？）
- **キャンセル理由** — どの理由が増加しているか？
- **引き止めオファーの種類** — どのオファーがどのセグメントに最も効果的か？

### キャンセルフローのA/Bテスト

一度に1つの変数をテストする:

| テスト | 仮説 | 指標 |
|------|------|------|
| 割引率（20% vs 30%） | より高い割引の方が引き止め率が高い | 引き止め率、LTV影響 |
| 一時停止期間（1ヶ月 vs 3ヶ月） | より長い一時停止の方が復帰率が高い | 再有効化率 |
| アンケートの位置（オファー前 vs 後） | アンケート先行の方がオファーをパーソナライズできる | 引き止め率 |
| オファー提示方法（モーダル vs フルページ） | フルページの方が注目を集める | 引き止め率 |
| コピーのトーン（共感的 vs 直接的） | 共感的な方が摩擦が減る | 引き止め率 |

**キャンセルフロー実験の実施方法:** **ab-test-setup** スキルを使って統計的に厳密なテストを設計する。PostHogはキャンセルフロー実験に適している — フィーチャーフラグでサーバーサイドでユーザーを異なるフローに分割でき、ファネル分析でキャンセルフローの各ステップ（アンケート→オファー→受諾/拒否→確認）を追跡できる。セットアップは [PostHog連携ガイド](../../tools/integrations/posthog.md) を参照。

## よくある失敗

- **キャンセルフローがまったくない** — 即時キャンセルは損をする。シンプルなアンケート＋1つのオファーでも10〜15%を引き止められる
- **キャンセルを見つけにくくする** — 隠れたキャンセルボタンは怒りと悪いレビューを生む。多くの法域で簡単なキャンセルが要求されている（FTC Click-to-Cancelルール）
- **理由に関係なく同じオファーを出す** — 一律割引では「機能がない」や「使っていない」に対応できない
- **割引が深すぎる** — 50%超の割引はキャンセルと再加入を繰り返す習慣をつける
- **非自発的解約を無視する** — 全解約の30〜50%を占め、最も修正しやすい
- **ダニングメールがない** — 決済失敗がサイレントにアカウントをキャンセルするままにする
- **罪悪感を利用するコピー** — 「本当に私たちを捨てますか？」はブランド信頼を傷つける
- **引き止めオファーのLTVを追跡しない** — 30日後に解約した「引き止めた」顧客は本当には引き止めていない
- **一時停止が長すぎる** — 3ヶ月を超える一時停止は再有効化されにくい。上限を設ける
- **キャンセル後のパスがない** — 再有効化を簡単にしてウィンバックメールをトリガーする。解約したユーザーの一部は戻ってくる
