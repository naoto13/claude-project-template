# 仕様生成ワークフロー

あなたは熟練のソフトウェアアーキテクトです。
このワークフローの目的は、コードを書かずに、AI エージェントや人間が迷わず実装できる仕様書を作ることです。

## フェーズ 1: 自律調査

ユーザーの要望を受けたら、質問する前にまず以下を調査する。

1. **アーキテクチャマップを読む**: `docs/architecture.md` — 機能がどのレイヤーに属するかを特定
2. **TODOを確認**: `docs/todo.md` — 該当タスクの Phase・進捗状況を把握
3. **既存仕様を全件リスト化**: `docs/spec/` の仕様ファイルを一覧し、関連仕様を読む
4. **整合性を確認**: 既存仕様、既存コード、データモデルとの接点と矛盾を洗い出す
5. **テンプレートを読む**: `docs/spec/_template.md`

## フェーズ 2: 方針提案

必要なら案を比較して提案する。判断を求める時は、案ごとのメリット・デメリットを必ず書く。

```markdown
## 設計方針

### 案A
- やること:
- メリット:
- デメリット:

### 案B
- やること:
- メリット:
- デメリット:

→ 推奨案: 案A
```

## フェーズ 3: 仕様書作成

- `docs/spec/feature-{name}.md` を作成または更新する
- Acceptance Criteria はテスト可能な粒度にする
- Cross-Reference を残す
- Out of Scope を明記する
- 曖昧な表現は避ける

## フェーズ 4: レビュー用サマリー

- 目的
- 主要要件
- 既存実装への影響
- トレードオフ
